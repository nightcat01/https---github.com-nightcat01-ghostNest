# Boundary Principles

이 문서는 어느 프로젝트에서든 적용할 수 있는 책임 경계 원칙이다.

## 원칙

- 구현 전에 변경 영역을 먼저 분류한다.
- 실행에 반드시 필요한 규격과 편집 편의 기능을 구분한다.
- 특정 화면, 특정 데이터, 특정 서비스 편의를 공통 영역에 넣지 않는다.
- 공통 영역 변경은 마지막 선택으로 둔다.
- 새 추상화는 실제 중복이나 복잡도가 생겼을 때만 만든다.

## 공통 영역에 둘 수 있는 것

- 런타임에 반드시 필요한 타입
- 공통 실행 규칙
- 여러 기능이 공유하는 최소 API
- 대체 구현이 어려운 핵심 상태 흐름

## 확장 영역에 둘 수 있는 것

- 편집 화면
- 카탈로그와 설명 데이터
- 개발자 도구
- 데모 preset
- 특정 서비스 adapter
- 특정 도메인 UI

## 확인 질문

- 이 변경은 실행에 반드시 필요한가?
- 다른 프로젝트에서도 같은 방식으로 필요한가?
- 특정 화면의 편의를 위해 공통 영역을 바꾸는 것은 아닌가?
- 플러그인, devtool, config, data로 충분히 표현 가능한가?
